# 系统万能封包

**理论上，只要彻底剔除掉当前电脑的硬件专属驱动（特别是主板芯片组、硬盘控制器 AHCI/NVMe 的驱动），这个系统就可以直接打包，安装到其他任何型号的电脑上。**

不过，在实际操作中，光是“简单粗暴地删除驱动”是不够的，因为 Windows 底层还有硬件抽象层（HAL）以及大量的注册表硬件残留。为了让这个系统到了新电脑上能**顺利开机且不蓝屏**，业界有一套标准、硬核的玩法。

## 🛠️ 官方的标准玩法：Sysprep（系统准备工具）

微软官方其实早就考虑到了这个需求（比如企业 IT 管理员需要给公司几百台不同型号的电脑装同一个配置好的系统）。Windows 系统里自带了一个深度剔除硬件信息的终极工具 —— **Sysprep**。

它可以一键将系统恢复到“开箱即用”状态（OOBE），并**彻底剥离所有当前电脑的硬件特征**。

### 具体操作步骤：

1. 在你已经配置好一切（装好了软件、汉化包、优化好设置）的系统上，按下 **`Win + R`**，输入 **`sysprep`** 回车。

2. 系统会自动打开一个文件夹，双击运行里面的 **`sysprep.exe`**。

   ```
   C:\Windows\System32\Sysprep\sysprep.exe
   ```

3. 在弹出的窗口中进行如下设置：

   - **系统清理操作：** 选择 **进入系统全新体验 (OOBE)**。
   - **通用：** **务必勾选“通用 (Generalize)”**（这一步就是最关键的“剔除驱动和硬件抽象层”）。
   - **关机选项：** 选择 **关机 (Shutdown)**。

4. 点击确定，系统会开始自动剥离硬件信息，完成后电脑会自动关机。

**注意：** 关机后，**千万不要再次开机进入这个系统！** 此时这个硬盘里的系统已经变成了“万能系统”。你可以用 PE U盘启动盘，把这个分区打包成 `WIM` 或 `GHO` 镜像，然后就可以把它安装到其他任何型号的电脑上了。

## 🔴 实际迁移中可能遇到的物理大坑

虽然原理可行，但在跨型号（尤其是跨世代、跨平台）迁移时，有三个致命的底层硬件冲突需要注意：

### 1. 硬盘模式冲突（最容易导致蓝屏 `INACCESSIBLE_BOOT_DEVICE`）

- 如果你原本的机器是一台老电脑，硬盘模式在 BIOS 里设置的是 **IDE** 或老旧的 **AHCI**。
- 当你把系统直接挪到一台现代的、使用 **NVMe M.2 固态硬盘** 的新电脑上时，新电脑由于找不到对应的 NVMe 控制器底层驱动，开机必然直接蓝屏。
- **解决办法：** 现在的第三方万能方案（如 IT天空的 EasySysprep 或 注入工具）会在系统封装时，强行往里面写入一套“万能磁盘驱动”，确保在新机器上能识别硬盘。

### 2. CPU 平台跨度过大（Intel 与 AMD 互换）

- 如果从 Intel 平台迁移到 AMD 平台，虽然 Windows 10/11 的自适应能力极强，但主板芯片组（Chipset）的底层调度逻辑完全不同。
- 如果不经过 Sysprep 清洗，直接热插拔硬盘过去，系统后台会疯狂报错，甚至无法加载核心电源管理。

### 3. 激活状态与数字权利

- Windows 的激活通常是绑定在“当前主板”上的。
- 当你把系统剔除驱动挪到新电脑上后，系统一旦联网检测到主板的 UUID 变了，**汉化和系统优化虽然还在，但系统会变成“未激活”状态**，需要重新输入密钥或重新激活。

## 💡 给你的折腾建议

如果你目前正打算把自己精心调教好、汉化好的 **Tiny10** 或者是某个配置好的系统，大范围复制到手里的其他老旧本子、医疗测试机或微型台式机上：

1. **不要手动去设备管理器里一个一个删驱动**，那样删不干净，且极易导致当前系统直接崩溃。
2. **首推使用 Sysprep 工具**（勾选“通用”）。
3. 如果面对的是配置差异极大的老机器（比如有些带 MXM 独显，有些是 Core M 无风扇），建议在用镜像把系统恢复到新电脑后，**第一次开机前，用 PE 里面的“离线驱动导入工具”（如 SkyIAR）**，提前把万能磁盘驱动打进去。这样可以保证 100% 顺利亮机进入桌面，进去之后再让系统联网自动补全特定显卡或声卡的驱动即可。

```

2026-05-30 00:36:13, Error                 SYSPRP Exit code of RemoveAllApps thread was 0x3cf2.

2026-05-30 00:36:13, Error                 SYSPRP ActionPlatform::LaunchModule: Failure occurred while executing 'SysprepGeneralizeValidate' from C:\Windows\System32\AppxSysprep.dll; dwRet = 0x3cf2

2026-05-30 00:36:13, Error                 SYSPRP SysprepSession::Validate: Error in validating actions from C:\Windows\System32\Sysprep\ActionFiles\Generalize.xml; dwRet = 0x3cf2

2026-05-30 00:36:13, Error                 SYSPRP RunPlatformActions:Failed while validating Sysprep session actions; dwRet = 0x3cf2

2026-05-30 00:36:13, Error      [0x0f0070] SYSPRP RunDlls:An error occurred while running registry sysprep DLLs, halting sysprep execution. dwRet = 0x3cf2
```

这个错误是非常经典的 **UWP (AppX) 应用不一致错误**。

具体原因是因为你在系统里使用了 **WinGet（Windows 软件包管理器）**。WinGet 在运行或自动更新时，在当前用户下注册了一个名为 `Microsoft.Winget.Source` 的数据源组件包。但因为它是**仅针对你当前登录的用户安装的，没有在全系统（All Users）中进行全局预配（Provisioned）**，Sysprep 在安全审查时认为这会导致生成的万能镜像在新电脑上损坏，所以直接罢工。

解决这个问题不需要补充任何组件，只需要在底层把这个碍事的 WinGet 遗留包**彻底拔除**。请按照以下步骤操作：

### 第一步：使用 PowerShell 强行卸载冲突包

1. 右键点击屏幕左下角的开始按钮，选择 **“Windows PowerShell (管理员)”**（务必是以管理员身份运行）。

2. 复制并粘贴以下命令，然后按**回车**：

   ```PowerShell
   Get-AppxPackage -AllUsers *Winget.Source* | Remove-AppxPackage -AllUsers
   ```
   
3. *备用强力命令：* 如果上面这一条命令运行后没有报错但心里不踏实，可以针对日志里那个精确的包名执行定点清除，复制并运行这行：

   ```PowerShell
   Remove-AppxPackage -Package "Microsoft.Winget.Source_2026.529.911.31_neutral__8wekyb3d8bbwe" -AllUsers
   ```

### 第二步：清理可能残留的预配包（安全预防）

有时候光删用户包还不够，如果它在全局预配列表里挂了名，也要顺手抹掉。在同一个 PowerShell 窗口中继续运行：

```PowerShell
Get-AppxProvisionedPackage -Online | Where-Object {$_.PackageName -like "*Winget.Source*"} | Remove-AppxProvisionedPackage -Online
```

### 第三步：检查并删除可能存在的其他冲突用户

Tiny10 有时候在安装或精简过程中会建立临时的审计账户或遗留账户，如果刚才的应用在别的隐蔽账户里也装了，Sysprep 还是会报错。

1. 在 PowerShell 中运行命令查看系统中的所有用户账户：

   ```powershell
   Get-UserProfile
   ```
   
2. 如果发现除了你当前使用的账户和 Administrator 之外，还有类似于 `UpdatusUser`、`Test` 或其他奇怪的 SID 目录，建议去“控制面板 -> 用户账户”里，把这些多余的无用账户**连同它们的文件一起彻底删除**。

### 第四步：重启并重新执行 Sysprep

1. 清理完成后，**务必重启一次电脑**，让底层的 AppX 服务彻底刷新缓存。
2. 重新按 `Win + R` 输入 `sysprep`，再次勾选 **OOBE**、勾选 **通用 (Generalize)**、选择 **关机**，点击确定。

这次它应该就能顺利通过 `AppxSysprep.dll` 的验证，完成驱动剥离并自动关机了！关机后记得直接用 PE U盘去打包备份 C 盘。

这正是你之前为了让英文版 Tiny10 变中文，下载并安装的 **简体中文本地体验包（Language Experience Pack）**。它在现代 Windows 10 中是以 UWP（应用商店）组件的形式存在的。由于你当时是在当前登录的用户账户下安装了它，而 Tiny10 底层没有将其“预配（Provision）”给全系统所有用户，Sysprep 的死板校验机制再次判定其为“不合规组件”，直接终止了进程。

### 💡 核心矛盾与两难境地

这是一个非常微妙的冲突：

- **如果你直接用命令删掉它：** 系统可能会退回英文，甚至导致你辛苦搞好的中文汉化环境被破坏。
- **如果不删它：** Sysprep 就会永远卡在这里无法进行驱动剥离。

为了保留汉化成果的同时顺利通过 Sysprep，我们需要采用“先全局预配，如果不行再强制剥离，到新机器再补”的极客应对策略。请按照以下步骤尝试：

### 方案一：强行将该中文包提升为“全系统全局预配”（首选推荐 ⭐⭐⭐⭐⭐）

既然 Sysprep 抱怨它“没有 provisioned for all users”，那我们就用命令在底层把它变成全局预配，堵住 Sysprep 的嘴。

1. 右键点击开始按钮，选择 **“Windows PowerShell (管理员)”**。

2. 复制并运行以下命令（直接利用系统里现有的缓存进行全局注册）：

   ```PowerShell
   Get-AppxPackage -AllUsers *LanguageExperiencePackzh-CN* | ForEach-Object {Add-AppxProvisionedPackage -Online -PackagePath $_.InstallLocation -SkipLicense}
   ```
   
3. 运行后如果屏幕没有红字报错，说明成功升级为全局组件。此时**立刻重新运行 `sysprep.exe`** 尝试封装。

### 方案二：如果方案一报错，只能在当前“母体”中强行拔除它

如果 Tiny10 因为精简了太多组件导致方案一无法全局注册，为了能把系统顺利打包带走，我们只能选择在封装前**暂时强行卸载**这个中文包，等系统到了新电脑上、开机进入桌面后，再重新执行一次 `dism` 离线汉化。

1. 以**管理员身份**打开 **PowerShell**。

2. 复制并粘贴以下命令，把这个阻碍封装的特定中文包彻底抹去：

   ```powershell
   Get-AppxPackage -AllUsers *LanguageExperiencePackzh-CN* | Remove-AppxPackage -AllUsers
   ```
   
3. 为了确保干净，针对日志里的精确包名再定点清除一次：

   ```powershell
   Remove-AppxPackage -Package "Microsoft.LanguageExperiencePackzh-CN_19041.81.277.0_neutral__8wekyb3d8bbwe" -AllUsers
   ```
   
4. **彻底清洗残留：**

   ```powershell
   Get-AppxProvisionedPackage -Online | Where-Object {$_.PackageName -like "*LanguageExperiencePackzh-CN*"} | Remove-AppxProvisionedPackage -Online
   ```
   
5. 清理完后，**重启电脑**，然后再次运行 `sysprep.exe`。此时没有了 UWP 中文包的阻碍，系统就能顺利剥离驱动并关机了。

### 🛠️ 方案二的后续（到了新电脑上怎么变回中文？）

如果你被迫选了方案二，当你把镜像还原到其他型号的新电脑、第一次开机释放完系统后，系统界面可能会暂时恢复成英文（或者部分中英交叠）。

不用担心，因为你手头已经有了之前下载好的官方原版 **`lp.cab`** 离线语言包：

1. 在新电脑上，把 `lp.cab` 放到 C 盘根目录。

2. 再次用管理员身份打开命令行运行：

   DOS

   ```
   dism /online /add-package /packagepath:C:\lp.cab
   ```

3. 重启新电脑，系统就又会彻底变回完美的简体中文了。

这就是折腾精简系统最有趣也最耗脑筋的地方——像打地鼠一样，出一个错，解一个逻辑。你可以先试一下方案一，看看能否直接瞒天过海！



# 避免用户重新登录

Windows 11 在通过 Sysprep 封装（尤其是使用了 `/generalize` 参数）后，系统会进入 **OOBE（开箱即用体验）** 阶段。在默认情况下，OOBE 阶段会强制你走完初始化流程，包括连接网络、登录或创建一个新的本地/微软账户，这就导致了你看到的“老用户还在，但被逼着又建了一个新用户”的情况。

要解决这个问题，实现**直接进入系统并使用老用户（或者直接自动登录），不提示创建新用户**，最标准且优雅的解决方案是**使用无人值守应答文件（`unattend.xml`）**。

下面是具体的实现方法和步骤：

## 核心思路：通过应答文件跳过 OOBE 创建用户阶段

我们需要制作一个 `unattend.xml` 文件，在里面配置两件事：

1. **自动跳过 OOBE 的用户创建界面（SkipUserOOBE）**。
2. **设置系统自动登录到你指定的那个老用户（AutoLogon）**。

### 第一步：准备应答文件（unattend.xml）

创建一个文本文件，将其命名为 `unattend.xml`，然后将以下代码复制进去。

> ⚠️ **注意修改**：请将代码中 `你的老用户名` 和 `你的密码` 替换为你封装前系统里那个老用户的实际账号和密码。

XML

```xml
<?xml version="1.0" encoding="utf-8"?>
<unattend xmlns="urn:schemas-microsoft-com:unattend">
    <settings pass="oobeSystem">
        <component name="Microsoft-Windows-Shell-Setup" processorArchitecture="amd64" publicKeyToken="31bf3856ad364e35" language="neutral" versionScope="nonSxS" xmlns:wcm="http://schemas.microsoft.com/WMIConfig/2002/State" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">
            <AutoLogon>
                <Password>
                    <Value></Value>
                    <PlainText>true</PlainText>
                </Password>
                <Enabled>true</Enabled>
                <LogonCount>1</LogonCount>
                <Username>VIP</Username>
            </AutoLogon>
            <OOBE>
                <HideEULAPage>true</HideEULAPage>
                <HideLocalAccountScreen>true</HideLocalAccountScreen>
                <HideOnlineAccountScreen>true</HideOnlineAccountScreen>
                <HideWirelessSetupInOOBE>true</HideWirelessSetupInOOBE>
                <NetworkLocation>Home</NetworkLocation>
                <ProtectYourPC>1</ProtectYourPC>
                <SkipMachineOOBE>true</SkipMachineOOBE>
                <SkipUserOOBE>true</SkipUserOOBE>
            </OOBE>
        </component>
    </settings>
</unattend>
```

### 第二步：将应答文件放入指定目录

把修改好的 `unattend.xml` 文件复制到原电脑的以下路径中： `C:\Windows\System32\Sysprep\`

### 第三步：使用命令行执行 Sysprep

不要直接双击打开 Sysprep 的图形界面，而是使用**管理员身份**打开 CMD（命令提示符），运行以下命令来调用应答文件进行封装：

```
cd C:\Windows\System32\Sysprep
sysprep.exe /generalize /oobe /shutdown /unattend:unattend.xml

```

## 封装后的实际效果

当你把这个封装好的镜像注入到其他电脑并开机时：

1. 系统会正常识别新电脑的硬件并安装基础驱动（Getting devices ready...）。
2. 当进入到原本该让你选择语言、联网、建用户的 OOBE 界面时，系统会读取 `unattend.xml` 中的指令，**直接隐式跳过**这些界面。
3. 系统会根据配置，**自动登录到你指定的老用户**中。

## 💡 补充避坑小贴士

1. **Windows 11 的联网强制要求**：Windows 11 在 OOBE 阶段默认强制联网。上面的应答文件已经尽量做了绕过，但为了万无一失，新电脑在首次开机部署时，**建议先不要插网线或连 Wi-Fi**，让它纯本地跑完第一次自动化登录。
2. **密码为空的情况**：如果你的老用户没有设置密码，在 `<Password>` 标签里保持空白即可（即 `<Value></Value>`），并将 `<PlainText>` 设为 `true`。
3. **安全考量**：应答文件中的密码虽然可以加密，但如果是明文（`PlainText: true`），部署完成后建议去 `C:\Windows\System32\Sysprep\` 下把这个 `unattend.xml` 删掉，防止密码泄露。



# 如何避免蓝屏

Windows系统封装（Sysprep）后，部署到其他硬件不一致的电脑上时，最常见的蓝屏（BSOD）错误码通常是 **`STOP: 0x0000007B`**（Inaccessible Boot Device），这通常是由于目标电脑的硬盘控制器驱动（SATA AHCI/RAID/NVMe）与原封装系统不匹配导致的。

要彻底解决由于缺少驱动导致的蓝屏问题，通常需要在封装阶段（事前预防）**和**部署/PE阶段（事后补救）双管齐下。以下是主流且高效的几种解决方案：

## 一、 事前预防：在系统封装时集成万能驱动

这是最标准的做法。在原机进行 `Sysprep` 之前或期间，通过工具将主流的硬件驱动（尤其是存储控制器和网卡驱动）预埋进系统镜像中。

### 1. 使用专业的封装辅助工具

原生 Windows 的 Sysprep 策略比较保守。建议使用国内成熟的系统封装助手（如 **IT天空的 Easy Sysprep**、**SysCeo的系统封装首席执行官** 等）。

- **原理**：这些工具支持在封装的各个阶段（如部署前、部署中、首次进系统前）调用**万能驱动（如驱动总裁、DrvCeo）**。
- **操作**：在封装工具中设置“部署中”或“首次进系统前”自动运行无人工干预的驱动安装包。当系统在目标机上第一阶段解压完成后，会自动扫描并注入匹配的离线驱动。

### 2. 使用 Dism+ 提前注入离线驱动

如果你喜欢纯净封装，不希望使用第三方封装软件，可以在封装前利用微软自带的 `DISM` 工具，将目标机群可能用到的 **NVMe/AHCI 驱动（即所谓的"磁盘控制器驱动"）** 提前整合进系统。

DOS

```
Dism /Image:C:\ /Add-Driver /Driver:D:\DriversFolder /Recurse
```

*注：此命令也可以在 PE 系统下，对已经恢复到 C 盘但尚未开机的系统镜像进行注入。*

## 二、 事后补救：在 PE 部署阶段动态注入驱动

如果系统已经封装完成，并且你已经在目标电脑上释放了镜像（此时开机蓝屏），**千万不要重新封装**，可以直接在 PE 环境下进行驱动注入。

### 1. 使用 PE 内的“离线驱动注入”工具

绝大多数装机维护盘（如优启通、微PE、无忧等）的 PE 系统中，都自带 **“离线驱动注入”** 或 **“SkyIAR”** 等工具。

- **操作步骤**：
  1. 将系统镜像释放到目标盘（C盘）。
  2. 不要重启，打开 PE 里的驱动注入工具。
  3. 目标路径选择刚刚释放好系统的 C 盘。
  4. 勾选 **AHCI/RAID/NVMe 驱动**（如果工具自带离线驱动包），或者手动指定目标机器官方下载的磁盘驱动解压文件夹。
  5. 点击“开始注入”，完成后再重启电脑。系统就能正常识别硬盘并开机了。

### 2. 检查并切换 BIOS/UEFI 硬盘模式

有时候不是缺驱动，而是原系统和目标机的硬盘工作模式不一致：

- 如果封装的系统是在 **AHCI** 模式下做的，而目标机 BIOS 里开启了 **Intel RST (RAID On) / VMD** 技术，大概率会蓝屏。
- **解决方法**：进目标机的 BIOS 设置，找到 `Storage` 或 `SATA Configuration`，将 `RAID/VMD` 模式临时修改为 `AHCI` 模式（或者反过来），通常即可顺利进系统，进系统后再补装对应的官方驱动。

## 三、 现代系统（Win10/Win11）的特殊建议

如果你封装的是 Windows 10 或 Windows 11，它们对现代硬件（如 NVMe 固态硬盘）的自带驱动支持已经非常完善，通常很少因为缺 NVMe 驱动蓝屏。如果依然蓝屏，重点排查：

1. **Intel VMD 驱动缺失**：第11代及以上酷睿处理器的笔记本/台式机，默认开启 VMD。必须在 PE 阶段向 C 盘注入 **Intel RST VMD 驱动**，否则必蓝屏 7B。
2. **电源管理/ACPI 冲突**：如果是从极老的平台（如 4 代酷睿）封装镜像去装最新的平台，由于 ACPI 规范升级，可能引发蓝屏。建议在封装时，确保原系统的电源管理驱动使用的是微软通用驱动。

## 💡 总结最佳工作流

> **最稳妥的装机流程**： 在 PE 下用常规工具（如 WinNTSetup）把系统写入目标 C 盘 ➔ **不要重启** ➔ 打开离线驱动注入工具（如驱动总裁/SkyIAR）➔ 扫描并注入目标机型的存储与网卡驱动 ➔ 重启电脑 ➔ 顺利进入系统部署界面。



# 应答文件解决驱动

在 Windows 系统封装（Sysprep）中，通过 `unattend.xml`（应答文件）来指定驱动程序路径，是实现自动化部署、彻底解决异构硬件蓝屏问题的最标准做法。

微软在应答文件中提供了专门的组件来指定离线驱动驱动库（Driver Paths）的搜索路径。在系统部署的 **`windowsPE`** 阶段或 **`offlineServicing`** 阶段，安装程序会自动扫描这些目录并安装匹配的驱动。

以下是具体的设置方法和 XML 代码示例：

## 核心设置：使用 `Microsoft-Windows-PnpCustomizationsNonWinPE`

最常用的方式是在 **`offlineServicing`** 阶段（离线服务阶段）配置 `Microsoft-Windows-PnpCustomizationsNonWinPE` 组件。这一阶段会在系统首次重启、进入桌面之前，把驱动注入到目标系统盘中。

### 1. 应答文件 XML 代码示例

你可以在你的 `unattend.xml` 中加入以下片段（注意根据你的目标系统架构选择 `processorArchitecture` 为 `amd64` 或 `x86`）：

XML

```
<?xml version="1.0" encoding="utf-8"?>
<unattend xmlns="urn:schemas-microsoft-com:unattend">
    <settings pass="offlineServicing">
        <component name="Microsoft-Windows-PnpCustomizationsNonWinPE" processorArchitecture="amd64" publicKeyToken="31bf3856ad364e35" language="neutral" versionScope="nonSxS" xmlns:wcm="http://schemas.microsoft.com/WMIConfig/2002/State" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">
            <DriverPaths>
                <PathAndCredentials wcm:action="add" wcm:keyValue="1">
                    <Path>C:\Drivers\Storage</Path>
                </PathAndCredentials>
                <PathAndCredentials wcm:action="add" wcm:keyValue="2">
                    <Path>C:\Drivers\Network</Path>
                </PathAndCredentials>
                <PathAndCredentials wcm:action="add" wcm:keyValue="3">
                    <Path>D:\Drivers</Path>
                </PathAndCredentials>
            </DriverPaths>
        </component>
    </settings>
</unattend>
```

### 2. 关键参数解析

- **`Path`**:
  - 指定驱动程序（`.inf` 文件）所在的文件夹。
  - **重要细节**：Windows 的 PnP（即插即用）管理器在扫描该路径时，**默认会递归扫描该文件夹下的所有子文件夹**。因此，你只需要指定一个总的驱动母目录（例如 `C:\Drivers`），它就会自动寻找里面所有的子目录。
- **`wcm:keyValue`**:
  - 用于区分多个路径的唯一键值，从 `1` 开始递增即可。
- **驱动格式要求**：
  - 这里的路径**必须指向解压后的驱动程序（包含 `.inf`、`.sys`、`.cat` 文件）**。直接放一个 `.exe` 或 `.msi` 的驱动安装包是无效的。

## 进阶：如何配合封装流程使用？

在实际生产环境中，为了让 `unattend.xml` 中的驱动路径生效，通常有两种配合玩法：

### 玩法 A：本地预埋驱动流（最推荐）

1. 在封装系统（Master Image）时，在 C 盘根目录创建一个文件夹（如 `C:\Windows\Inf_Drivers`）。
2. 在 `unattend.xml` 中将路径指定为 `C:\Windows\Inf_Drivers`。
3. 执行 Sysprep 封装。
4. **在目标机部署时**：在 PE 环境下，先把目标机型特有的驱动（或万能驱动包）释放到目标盘的 `C:\Windows\Inf_Drivers` 目录中，然后再引导开机。
5. 系统一开机，应答文件生效，自动从该目录抓取存储/网卡驱动，完美进系统。

### 玩法 B：网络共享注入流（适合企业批量部署）

如果目标机群在部署时已经可以通过网络连接到服务器，可以直接填写 UNC 网络路径。

XML

```
<PathAndCredentials wcm:action="add" wcm:keyValue="1">
    <Path>\\DeploymentServer\Share\Drivers\Lenovo_M70t</Path>
    <Credentials>
        <Domain>DomainName</Domain>
        <Username>Administrator</Username>
        <Password>YourPassword</Password>
    </Credentials>
</PathAndCredentials>
```

## 🛠️ 强烈建议：使用 WSIM 工具来生成

纯手写 XML 容易因为标签闭合、大小写或 `publicKeyToken` 错误导致 `unattend.xml` 解析失败报错。

建议下载微软官方的 **Windows ADK (Deployment Tools)**，使用其中的 **WSIM (Windows System Image Manager，Windows 系统映像管理器)**：

1. 打开你的系统镜像（`install.wim`）生成目录文件。
2. 在左下角的“组件”中找到 `Microsoft-Windows-PnpCustomizationsNonWinPE`。
3. 右键选择“添加到阶段 5 offlineServicing”。
4. 在右侧的 `DriverPaths` 上右键选择“新建 PathAndCredentials”。
5. 直接在属性面板的 `Path` 栏输入你的驱动路径。
6. 保存生成的 `unattend.xml` 即可。



# 指令注入驱动

系统封装（Sysprep）完成后，在将镜像恢复（注入）到目标电脑时，如果目标电脑存在硬件差异（如新的网卡、RAID控制器等），确实需要使用 **DISM (Deployment Image Servicing and Management)** 命令行工具来注入驱动，以确保系统能正常启动和运行。

根据你在部署系统时的不同阶段，主要有以下**两种操作场景**。请确保你手里有目标电脑的**纯驱动文件**（包含 `.inf`、`.sys`、`.cat` 等文件的文件夹，而不是 `.exe` 安装包）。

## 场景一：在 Windows PE 环境下注入（最推荐、最常用）

当你用 U 盘启动盘进入 PE，并使用 WinNTSetup 或 CGI 等工具把系统镜像（`.wim` 或 `.esd`）恢复到目标电脑的 C 盘后，**先不要重启**，直接在 PE 里打开**命令提示符 (CMD)**。

### 1. 确认盘符

在 PE 系统中，目标电脑的 C 盘盘符可能会发生改变（例如变成 D 盘或 E 盘）。请先在资源管理器里确认**系统实际被恢复到了哪个盘符**，以及你的**驱动文件夹存放在哪里**。

### 2. 执行 DISM 命令

打开 PE 里的命令提示符（以管理员身份），运行以下命令：

DOS

```
dism /Image:C:\ /Add-Driver /Driver:D:\Drivers /Recurse
```

**参数详细拆解：**

- `/Image:C:\` ：指定你刚才恢复好的、**目标系统所在的盘符**（如果是 D 盘就写 `D:\`）。
- `/Add-Driver` ：告诉 DISM 这是要添加驱动。
- `/Driver:D:\Drivers` ：指定你的**驱动程序所在文件夹路径**（请根据实际情况修改）。
- `/Recurse` ：**非常重要！** 意思是**递归查询**，DISM 会自动检索该文件夹及其所有子文件夹下的所有 `.inf` 驱动文件。

> 💡 **提示：** 如果只想注入某一个特定的驱动，可以直接指向该 `.inf` 文件，例如：`/Driver:D:\Drivers\Network\net.inf`。

## 场景二：在系统部署（Mini-Setup）阶段自动注入

如果你希望在系统恢复完成后、首次开机进入桌面的那个“黑屏/蓝屏滚动条”部署阶段自动完成注入，可以利用系统封装时生成的 **`Unattend.xml`（无人值守应答文件）**。

### 1. 修改应答文件

在应答文件中，利用 `Microsoft-Windows-PnpCustomizationNonWinPE` 组件，在 **`offlineServicing`** 阶段添加驱动路径。

XML

```
<settings pass="offlineServicing">
    <component name="Microsoft-Windows-PnpCustomizationNonWinPE" processorArchitecture="amd64" publicKeyToken="31bf3856ad364e35" language="neutral" versionScope="nonSxS" xmlns:wic="http://schemas.microsoft.com/appx/2010/manifest">
        <DriverPaths>
            <PathAndCredentials wcm:action="add" wcm:keyValue="1">
                <Path>C:\Drivers</Path> 
            </PathAndCredentials>
        </DriverPaths>
    </component>
</settings>
```

### 2. 配合本地方案

配合这个方案，你通常需要在镜像恢复到 C 盘时，顺便把驱动文件夹拷贝到 C 盘某个固定目录（如 `C:\Drivers`），系统在首次开机离线配置时就会自动调用 DISM 扫描并安装该目录下的驱动。

## ⚠️ 避坑指南与核心技巧

1. **驱动格式必须解压：** DISM 无法识别 `.exe` 或 `.msi` 的驱动安装包。如果是官方下载的安装包，请先用 7-Zip 等工具将其解压，提取出含有 `.inf` 的标准驱动目录。

2. **驱动签名问题（针对 Win10/Win11）：** 如果你要注入的是某些魔改驱动或未经过微软数字签名的驱动，DISM 可能会报错拒绝注入。此时需要加上 `/ForceUnsigned` 参数：

   DOS

   ```
   dism /Image:C:\ /Add-Driver /Driver:D:\Drivers /Recurse /ForceUnsigned
   ```

3. **精准注入 vs 盲目全注：** 如果是给特定电脑万能部署，建议只注入网卡（LAN/WLAN）**和**磁盘控制器（NVMe/RAID）驱动。显卡、声卡等大体积驱动建议留到进桌面后再由驱动总裁或 IT sky 等万能驱动工具处理，盲目用 DISM 注入数 GB 的显卡驱动会导致部署过程极慢，甚至引起蓝屏。
